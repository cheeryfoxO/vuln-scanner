## 二、挖洞完整流程（5 步）

### Step 1：选择目标

| 平台 | 说明 |
|------|------|
| **HackerOne** | 注册账号 → Programs → 筛选 `response_disclosure` 或 `none` 策略的程序 → 这些竞争最少 |
| **Bugcrowd** | 同上，找低优先级的公开程序 |
| **自有目标** | ⚠️ 必须有书面授权，否则违法 |

**新手建议：** 找 HackerOne 上 VDP（无赏金）程序练手，流程跑通后再接有赏金的。

### Step 2：信息收集（1~3 分钟）

对目标只跑信息收集模块，不触发 WAF：

```bash
# 基础信息收集
python -m scanner scan "https://target.com" \
  -m subdomain,dirscan,params,headers,cors \
  -v --delay 100 \
  -o recon.json -r recon.html
```

这一步会得到：
- **subdomain**：子域名列表（183 词表探测）
- **dirscan**：敏感文件/目录（324 词表，自动内容指纹识别）
- **params**：页面上的输入参数（form、URL 参数、JS 端点）
- **headers**：缺失的安全头
- **cors**：CORS 配置问题

**产出：** `recon.html` 用浏览器打开，按严重级别分组的报告

### Step 3：逐参数精准打击（核心步骤）

**不要用 `-m all` 扫整个域名！** 每发一个 payload 都可能触发 WAF。正确做法：

```bash
# 假设 Step 2 发现这些参数点：
#   /search?q=xxx      → 可能 XSS、SQLi、SSTI
#   /user?id=123       → 可能 SQLi、LFI
#   /api/ping?host=x   → 可能 CMDi、SSRF
#   /redirect?url=xxx  → 可能 Open Redirect

# 逐个参数点测试：
python -m scanner scan "https://target.com/search?q=test" \
  -m sqli,xss,ssti --delay 200 -v -o search.json

python -m scanner scan "https://target.com/user?id=1" \
  -m sqli,lfi --delay 200 -v -o user.json

python -m scanner scan "https://target.com/api/ping?host=127.0.0.1" \
  -m cmdi,ssrf --delay 200 -v -o ping.json

python -m scanner scan "https://target.com/redirect?url=/" \
  -m redirect --delay 200 -v -o redirect.json
```

**为什么逐参数？** WAF 可能对 `/search` 很严（面向用户）但对 `/api` 很松（内部接口）。

### Step 4：验证发现（人工确认）

工具标记的内容 ≠ 有效漏洞。逐条验证：

```bash
# 工具说 dirscan 找到的，手工确认
curl -s "https://target.com/.env" | head -20
# 如果看到 DB_PASSWORD=xxx → 截图，记下来

curl -s "https://target.com/.git/config"
# 如果有 [remote "origin"] → 用 git-dumper 下载

# 工具说 sqli 的，手工确认
curl "https://target.com/search?q=test'+OR+1=1--"
curl "https://target.com/search?q=test'+OR+1=2--"
# 响应长度不同 → 可能真存在，用 sqlmap 深入

# 工具说 xss 的，手工确认
# 浏览器访问 https://target.com/search?q=<script>alert(1)</script>
# 弹框 → 真的
```

**验证清单：**
- [ ] 截图证明漏洞存在
- [ ] 录制操作步骤
- [ ] 确认危害范围（影响哪些用户/数据）

### Step 5：写报告

工具生成的 HTML 报告是参考，不是直接提交的。提交给平台需要：

```
标题：[目标名] Reflected XSS in /search parameter

严重级别：High (P2)

描述：
The `q` parameter in `/search` endpoint reflects user input
without sanitization, allowing arbitrary JavaScript execution.

复现步骤：
1. 访问 https://target.com/search?q=<script>alert(document.cookie)</script>
2. 观察弹框显示当前域名下的 cookie

影响：
攻击者可以通过钓鱼链接窃取用户 session、重定向到钓鱼页面、
或修改页面内容。

修复建议：
对 `q` 参数做 HTML 实体编码后再输出到页面。
```

## 三、各模块详解与使用场景

### 3.1 信息收集类

#### subdomain — 子域名枚举
```bash
python -m scanner scan "example.com" -m subdomain -v
```
**价值：** 子域名往往指向测试环境、旧版本、管理后台——防护更弱。
**注意：** 183 词表是基础版，建议后续换大词表。

#### dirscan — 敏感文件探测
```bash
python -m scanner scan "https://target.com" -m dirscan -v
```
**自动做的事：**
1. HEAD 探测 324 个路径（.env, .git/config, wp-config.php.bak, id_rsa...）
2. 返回 200 的自动 GET 内容并做指纹匹配（DB_PASSWORD、BEGIN PRIVATE KEY、AWS 密钥）
3. 匹配到的自动提升严重级别

**最高价值发现：** `.env`（含密钥）、`.git/config`（可重建仓库）、`backup.sql`（数据库泄露）

#### params — 参数提取
```bash
python -m scanner scan "https://target.com/page" -m params -v
```
**输出：** 页面上的 `<form>`、`<input>`、URL 参数、`<script>` 中的 API 端点。
后面注入测试就基于这些参数。

### 3.2 注入类（误报多，必须人工确认）

#### sqli — SQL 注入（critical）
```bash
python -m scanner scan "https://target.com/page?id=1" -m sqli -v --delay 200
```
**检测方式：**
- 阶段 1：错误回显（发送 `'` `"` 等，检查响应是否含 SQL 错误信息）
- 阶段 2：时间盲注（发送 `sleep(n)` payload，检测响应延迟）

**标记为 critical 的条件：** 错误回显直接成功 → 几乎肯定有注入。

#### xss — 反射 XSS（high）
```bash
python -m scanner scan "https://target.com/search?q=test" -m xss -v --delay 200
```
**检测方式：** 注入 `<script>alert(1)</script>`、`<img onerror=...>` 等，检查响应 HTML 中是否完整反射。
**注意：** WAF 最常拦截 XSS payload，配合 `--delay 300` 降低速率。

#### dom_xss — DOM XSS（high）
对需要爬虫配合的页面使用。

#### stored_xss — 存储 XSS（high）
自动填充表单 → 提交 → 重新访问检查是否存留。

#### cmdi — 命令注入（critical）
```bash
python -m scanner scan "https://target.com/ping?host=127.0.0.1" -m cmdi -v --delay 200
```
**检测方式：**
- 阶段 1：错误输出检测（注入 `; id`、`| whoami`，检查响应中的 uid/gid 或 net user 输出）
- 阶段 2：时间盲注（注入 `sleep 5`，检测响应延迟）
- 带 5 种 WAF 绕过编码（url编码、大小写混合、双编码、空字节等）

#### lfi — 文件包含（high）
```bash
python -m scanner scan "https://target.com/page?file=index" -m lfi -v --delay 200
```
**检测 payload：** `../../../../etc/passwd`、`../../windows/win.ini`、`php://filter/...`

#### ssti — 模板注入（critical）
```bash
python -m scanner scan "https://target.com/preview?name=test" -m ssti -v --delay 200
```
**检测方式：**
- 阶段 1：表达式求值（`{{7*7}}` → 检查输出是否出现 `49`）
- 阶段 2：时间盲注（Jinja2 RCE gadget → `sleep 5`）

### 3.3 服务端类

#### ssrf — 服务端请求伪造（high）
```bash
python -m scanner scan "https://target.com/fetch?url=http://example.com" -m ssrf -v
```
**检测目标：** AWS 元数据（169.254.169.254）、本地服务、SSH banner。

#### redirect — 开放重定向（medium）
```bash
python -m scanner scan "https://target.com/redirect?url=/" -m redirect -v
```

### 3.4 配置/前端类（误报少，可直接提交）

#### csrf — CSRF 防护缺失（medium）
```bash
python -m scanner scan "https://target.com/settings" -m csrf -v
```
**检测：** 找 POST 表单 → 检查是否含 csrf_token / authenticity_token 等字段。

#### headers — 安全头缺失（low）
```bash
python -m scanner scan "https://target.com" -m headers -v
```
**检查 7 个头：** HSTS、CSP、X-Frame-Options、X-Content-Type-Options、Referrer-Policy、Permissions-Policy、ACAO 通配符。
**虽然严重级别低，但常被接受为有效报告。**

#### cors — CORS 配置错误（high）
```bash
python -m scanner scan "https://target.com" -m cors -v
```
**检测：** ACAO 是否反射请求中的 Origin，配合 ACAC: true 提升为 critical。

## 四、进阶用法

### 4.1 通过认证扫描

很多漏洞在登录后才暴露：

```bash
# 从浏览器复制登录后的 Cookie
python -m scanner scan "https://target.com/dashboard" \
  -m sqli,xss,csrf -v --delay 200 \
  --cookie "session=eyJ...; csrftoken=abc123"
```

### 4.2 自定义请求头

```bash
python -m scanner scan "https://target.com/api/v2" \
  -m sqli -v \
  --header "Authorization: Bearer eyJ..." \
  --header "X-API-Key: secret123"
```

### 4.3 爬虫 + 全模块扫描（慎用）

**仅对低风险的测试环境或有授权的自建靶场使用：**

```bash
# depth=2：爬种子页 + 一层链接
# scope：不跳出目标域名
# delay=200：减少被封概率
python -m scanner scan "https://test.target.com" \
  -m all --depth 2 --scope "*.target.com" --delay 200 \
  -v -o full_scan.json -r full_scan.html
```

**注意：** WAF 看到 200ms 间隔、大量异常 payload，还是会封 IP。

### 4.4 自定义词表

```bash
python -m scanner scan "https://target.com" \
  -m dirscan --dirscan-wordlist my_4000_dirs.txt \
  -m subdomain --subdomain-wordlist my_10k_subdomains.txt \
  -v
```

## 五、实战策略总结

| 阶段 | 命令 | 时间 |
|------|------|------|
| 信息收集 | `-m subdomain,dirscan,params,headers,cors` | 1~3 min |
| 爬虫扩展 | 加 `--depth 2 --scope *.target.com` | 2~5 min |
| 注入精准打击 | 逐参数 `-m sqli,xss,cmdi,lfi,ssti` | 每参数 1~2 min |
| 服务端测试 | `-m ssrf,redirect,csrf` | 1 min |
| 导出报告 | `-o scan.json -r report.html` | 瞬时 |

### 投入产出比排序（从高到低）

1. **dirscan** — 找到 .env/.git 就直接交，单人搞定，不需要复杂验证
2. **headers + cors** — 误报率最低，报告好写
3. **Open Redirect** — 检测简单，确认容易
4. **反射 XSS** — 常见但 WAF 拦截多，需要耐心
5. **SQLi** — 价值高但误报多，需要手工深入验证
6. **SSRF** — 罕见但一旦确认通常是 critical
7. **CMDi / SSTI** — 最稀有但也最高危

### 关键原则

```
工具 = 体力活加速器，不是人脑替代品

自动收集参数  →  手动判断哪个参数有意义
自动发 3000 payload  →  手动过滤 2995 个误报  
自动标记高危    →  手动验证是否真的高危
自动生成报告    →  手动写出能拿赏金的 PoC
```

## 六、常见问题

**Q: 扫自己本地靶场正常，扫线上目标什么都没有？**
A: WAF 拦截了。调高 `--delay` 到 500~1000ms，或换小众模块先试（headers、cors、csrf 不触发 WAF）。

**Q: 被封 IP 了怎么办？**
A: 用代理/VPN 换 IP 继续，或降低 `--delay` 到 0（但如果 WAF 封了就没办法）。

**Q: 工具扫出了一堆东西，不知道怎么验证？**
A: 优先验证 `critical` + `high`，看 evidence 字段。如果 evidence 显示 `SQL syntax error`，用 sqlmap 验证。如果 evidence 显示 `DB_PASSWORD`，curl 确认。

**Q: 提交给平台用什么格式？**
A: 每个平台都有模板。核心四要素：标题、复现步骤、影响、修复建议。附截图/PoC 视频。

**Q: 一个月能赚多少？**
A: 新手前 3 个月通常 0。入门后（3-6 月）平均每个月 $100~500。高手（1年+）能到 $2000~5000/月。这行是长期积累，没有捷径。
